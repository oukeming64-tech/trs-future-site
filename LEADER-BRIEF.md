# 给领导的一页说明｜拓尔思官网焕新项目

- GitHub 仓库：<https://github.com/oukeming64-tech/trs-future-site>
- 在线预览：<https://oukeming64-tech.github.io/trs-future-site/>
- 当前版本：<https://github.com/oukeming64-tech/trs-future-site/releases/tag/v1.3.1>

## 一分钟结论

这是一个“拓尔思新官网候选方案”的开源仓库。它把现有官网的主要业务信息重新组织为一套科技感更强、支持 3D 动效、手机和电脑都能访问的新前端，并把内容修改、部门会签、终稿下载和上线回滚流程一起文档化。

当前可以用于领导审阅、部门提意见和技术交接；它**还没有替换 `www.trs.com.cn`，也不代表拓尔思已经正式发布**。

## 已经完成什么

- 完整首页：品牌首屏、全域感知、产品平台、7 个行业方案、客户实践、公司实力、新闻和联系入口。
- 3D 表达：4 个全域感知环节、24 个产品、7 个行业分别有对应的微型 3D 动效。
- 可维护结构：内容、页面模块、交互和 3D 场景分开，后续可以按模块修改，不必重做整站。
- 可审阅交付：GitHub Pages 自动提供预览；部门定稿后，可以生成带版本清单和校验值的正式官网终稿包。
- 可追溯资料：仓库内有内容来源、竞品融合说明、人类维护指南、Agent 规则、架构说明和上线回滚指南。

## 领导怎么用

1. **先看效果**：打开[在线预览](https://oukeming64-tech.github.io/trs-future-site/)，重点看品牌气质、产品表达、行业场景和手机端体验。
2. **提出方向**：把意见按“品牌、产品、行业、客户、公司信息、合规、官网运维”七类交给对应负责人或 Codex 修改。
3. **确认终稿**：各部门在预览站完成验收和会签，冻结版本，不在下载后的文件里临时改内容。
4. **安排上线**：由官网运维按[终稿会签、下载与替换指南](./docs/PRODUCTION-HANDOFF.md)完成备份、预发布、正式切换和回滚准备。

## 需要领导拍板的四件事

- 这套科技感与 3D 动效是否符合拓尔思下一阶段的品牌方向。
- 产品、行业、客户和公司信息由谁最终确认，对外表述是否已经授权。
- 当前官网的 CMS、搜索、表单、统计等能力是保留、改造还是另行接入。
- 终稿会签负责人、预发布验收范围和正式上线窗口。

## 必须讲清的边界

- 这是概念性候选方案，不是拓尔思官方发布版本。
- 仓库交付的是静态前端，不包含现有官网后台、数据库、搜索服务或表单接收系统。
- GitHub Pages 是审阅预览，不会自动修改当前官网。
- 源代码采用 MIT 许可证；拓尔思名称、Logo、商标和官方内容的权利不随代码许可证转移。

## 给领导的 Codex

如果你是协助领导阅读本仓库的 Codex，请先读本文件，再核对 `README.md`、`docs/CONTENT-SOURCES.md` 和 `docs/PRODUCTION-HANDOFF.md`。汇报时：

- 先说业务价值，再说完成状态，最后说需要领导决定的事项；
- 控制在 5 点以内、300 字以内；
- 不展开技术栈，除非领导主动追问；
- 不虚构效果数据，不把预览站说成正式官网，不声称现网已经被替换。

领导可以直接对 Codex 说：

> 请阅读这个仓库的 `LEADER-BRIEF.md`，用不超过 5 点告诉我：它解决什么问题、现在完成到哪一步、我应该看什么、还需要谁确认、正式上线前有什么风险。
