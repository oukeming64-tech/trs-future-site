# 官网终稿会签、下载与替换指南

这份指南用于各部门完成修改并确认终稿之后，把已经批准的版本下载下来，交给官网运维人员替换当前官网。它不授权任何人跳过审批直接上线。

> 当前仓库仍是拓尔思官网焕新概念站。GitHub Pages 只用于公开预览；是否替换 `www.trs.com.cn`，必须由拓尔思品牌、业务、法务和官网运维负责人共同确认。

## 先分清两种成品

| 成品 | 用途 | 地址或文件 | 是否能直接替换官网 |
| --- | --- | --- | --- |
| GitHub Pages 预览版 | 部门审阅、手机和电脑验收 | `https://oukeming64-tech.github.io/trs-future-site/` | 不能，资源带 `/trs-future-site/` 子路径 |
| 正式官网终稿包 | 交给官网运维做预发布和正式切换 | Actions 中的 `trs-website-production-…` 压缩包 | 可以作为静态前端包，但必须先按本指南验收 |

正式包会使用官网根路径 `/`，并把页面 canonical、Open Graph 等站点元数据指向 `https://www.trs.com.cn/`。不要把 GitHub Pages 的构建产物复制到现网。

## 第一步：各部门修改完成后进行部门会签

每个部门只确认自己负责的事实和体验。任何一项仍为“待确认”，都不能生成终稿。

| 部门 | 必须确认的内容 | 建议验收人 |
| --- | --- | --- |
| 品牌 / 市场 | Logo、品牌名称、主视觉、口号、色彩、分享图和对外语气 | 品牌负责人 |
| 产品 / 研发 | 产品名称、能力说明、业务结果、产品链接，以及 24 套产品 3D 原型表达是否准确 | 产品负责人 |
| 行业方案 / 业务 | 7 个行业的价值主张、典型场景、行业链接和各自 3D 动效是否符合实际 | 行业负责人 |
| 销售 / 客户成功 | 客户名称、案例分类、合作表述与公开授权范围 | 销售或客户负责人 |
| 综合 / 公关 | 公司介绍、公司数据、新闻、里程碑、地址、电话和邮箱 | 公关负责人 |
| 法务 / 合规 / 董秘 | 商标、版权、数据真实性、法定披露、隐私与统计代码合规 | 法务或董秘负责人 |
| 信息化 / 官网运维 | 域名、HTTPS、现有 CMS 或服务器、CDN、缓存、安全头、备份与回滚条件 | 官网运维负责人 |

部门修改入口和模块位置见 [官网维护指南](./EDITING-GUIDE.md)。涉及公开事实时，同时核对 [内容来源](./CONTENT-SOURCES.md)。

## 第二步：冻结终稿

会签全部通过后，由仓库维护者完成以下动作：

1. 把所有批准的改动合并到 `main`。
2. 等待 `Deploy GitHub Pages` 成功，在电脑和手机上做最后一次预览验收。
3. 记录 Git 提交号、版本号、会签表和预计上线时间。
4. 给终稿一个不重复的标签，例如 `v1.3.1-final`。
5. 冻结后不再直接改文案或动画；新的修改重新走会签并生成新的终稿包。

如果概念站说明、非官方发布说明或品牌权利说明需要在正式官网移除，必须先得到品牌与法务的明确批准，再作为一次可追溯的源代码改动处理，不能在下载后的文件里临时删除。

## 第三步：生成并下载终稿包

需要仓库写入权限的维护者生成终稿包：

1. 进入 GitHub 仓库的 **Actions**。
2. 打开 **Build production handoff**。
3. 选择 **Run workflow**，分支选 `main`。
4. 在 `release_label` 填入冻结的终稿标签，例如 `v1.3.1-final`。
5. 点击 **Run workflow**，等待全部检查变为绿色。

### 下载终稿包

1. 打开刚刚成功的 workflow run。
2. 在页面底部找到 **Artifacts**。
3. 点击名称以 `trs-website-production-` 开头的文件下载 ZIP。
4. 解压后先阅读 `上线交接指南.md`，再检查 `handoff-manifest.json` 和 `checksums.sha256`；需要追溯改法或事实时，包内也有维护指南和内容来源副本。
5. 真正用于网站部署的是 `site/` **里面的内容**，不是外层终稿包，也不是 `site` 文件夹本身。

GitHub 官方说明：下载 Actions artifact 需要登录并拥有仓库读取权限；本工作流把终稿包保留 30 天。官方操作入口见 [下载 workflow artifacts](https://docs.github.com/en/actions/how-tos/manage-workflow-runs/download-workflow-artifacts?tool=webui) 和 [手动运行 workflow](https://docs.github.com/en/enterprise-cloud@latest/actions/how-tos/manage-workflow-runs/manually-run-a-workflow?apiVersion=2022-11-28&tool=webui)。

熟悉 GitHub CLI 的维护者也可以使用：

```bash
gh run download RUN_ID -n ARTIFACT_NAME
```

## 第四步：交付包验真

解压后的结构应为：

```text
终稿包/
├── site/                    # 只上传这个目录里的内容
│   ├── index.html
│   ├── 404.html
│   ├── _next/
│   ├── trs-logo.png
│   └── og.png
├── 上线交接指南.md          # 本指南的随包副本，不上传
├── handoff-manifest.json    # 版本、提交号、目标域名和构建时间
├── EDITING-GUIDE.md         # 本版本对应的模块维护指南，不上传
├── CONTENT-SOURCES.md       # 本版本对应的公开事实来源，不上传
└── checksums.sha256         # 除自身外，包内全部文件的完整性校验
```

打开 `handoff-manifest.json`，至少确认：

- `releaseLabel` 与会签记录一致；
- `sourceCommit` 与冻结的 `main` 提交一致；
- `targetUrl` 是 `https://www.trs.com.cn/`；
- `deploymentMode` 是 `static-site-root`；
- 版本号与本次发布记录一致。

官网运维可在终稿包根目录执行以下校验；全部显示 `OK` 才继续：

```bash
shasum -a 256 -c checksums.sha256
```

## 第五步：先在预发布环境替换

本仓库交付的是纯静态前端，不包含数据库、登录、搜索后端、表单接收、运行时 CMS 或投资者关系系统。官网运维必须先确认当前官网属于哪一种情况。

### 情况 A：现网允许整站静态文件替换

1. 备份当前网站根目录、服务器配置、证书配置和 CDN 规则。
2. 如果旧站仍有 CMS 或数据库，再分别备份程序、上传文件和数据库。
3. 把 `site/` 里面的内容上传到预发布环境的网站根目录；不要多套一层 `site/`。
4. 保持 HTTPS、压缩、MIME 类型、404 页面和静态缓存配置正确。
5. 按下方验收清单通过后，再安排现网窗口。

### 情况 B：现网依赖 CMS、反向代理或动态服务

不要直接覆盖服务器目录。由官网运维把 `site/` 作为新的前端层接入现有架构，并逐项确认旧站的搜索、表单、后台发布、投资者关系、统计和跳转是否仍由原服务承接。需要保留动态能力时，应先补接口或路由适配，再上线。

当前仓库无法从公开页面判断 `www.trs.com.cn` 的内部部署方式，因此最终采用 A 还是 B，只能由官网运维根据服务器和 CMS 实际情况决定。

## 第六步：预发布验收

至少完成以下检查：

- 首页、导航、页脚以及所有模块都能打开；刷新页面不出现 404。
- Logo、字体、CSS、JavaScript、分享图和 `_next/` 资源没有丢失。
- 4 个全域感知、24 个产品、7 个行业的 3D 场景能显示；WebGL 不可用时仍有降级画面。
- 产品、客户切换，移动菜单，键盘操作和减少动态效果模式正常。
- 手机、平板、电脑没有横向溢出，重要文字和按钮可读。
- 电话、邮箱、产品、行业、新闻、文档和投资者关系链接均指向已批准地址。
- 页面标题、描述、canonical 和分享卡片均指向正式域名。
- Cookie、访问统计、隐私文本与安全策略已经过法务和信息化确认。
- 404、HTTPS、缓存策略、CDN 刷新和监控告警已验证。
- 概念站说明是否保留，已经有品牌与法务的书面结论。

## 第七步：正式替换当前官网

1. 选择访问量较低且有人值守的上线窗口。
2. 再做一次现网完整备份，并记录备份位置和回滚负责人。
3. 将预发布验收通过的**同一个终稿包**部署到现网，不在服务器上临时改文件。
4. 刷新 CDN 和浏览器缓存，检查首页与关键链接。
5. 连续观察访问错误、资源 404、JavaScript 报错、加载性能和业务反馈。
6. 把上线时间、终稿标签、提交号、Actions run、操作人和验收结果写入发布记录。

## 回滚条件与步骤

以下任一情况出现时，先回滚，不在现网边修边试：

- 首页或主要栏目无法访问；
- CSS、JavaScript、Logo 或大批静态资源 404；
- 关键业务链接、合规入口或联系方式错误；
- 移动端严重不可用，或多数设备无法显示核心内容；
- 监控、网关、CDN 或安全策略出现异常；
- 部门发现终稿事实错误或未经授权内容。

回滚步骤：

1. 恢复上线前保存的旧站文件和服务器配置；动态站同时恢复对应程序与数据库版本。
2. 刷新 CDN 缓存并确认旧站首页和关键业务恢复。
3. 记录问题、终稿标签、提交号和影响范围。
4. 在源代码中修复并重新走测试、会签、冻结和终稿包生成流程；不要继续使用已经判定失败的压缩包。

## 责任边界

- GitHub Actions 只负责从已批准的源代码生成、测试和封装终稿包，不会自动操作 `www.trs.com.cn`。
- 仓库维护者对“源码与终稿包一致”负责；各部门对自己会签的事实负责；官网运维对服务器、CMS、备份、上线和回滚负责。
- 每次上线只接受由 **Build production handoff** 生成、清单可追溯且校验通过的包。
- `out/`、GitHub Pages artifact 或个人电脑上临时修改过的文件不能作为正式交付物。
